## 1. Distributed systems concepts this project demonstrates

### Eventual consistency
When a user registers, the account exists **immediately**, but the welcome notification appears
a moment later (after the event is consumed). The system is **eventually consistent** — the two
databases aren't updated in one atomic transaction. That's the deliberate trade-off for
decoupling: you give up immediate cross-service consistency to gain independence and resilience.

**Interviewer:** *"What if the NotificationService is down when the event is published?"*
**You:** "RabbitMQ holds the message in the queue. When NotificationService comes back up, MassTransit
delivers it and the consumer runs. No notification is lost — that's the durability benefit of a
broker over a direct HTTP call. With a synchronous HTTP call, if the downstream was down, the
whole registration could fail or block."

### Loose coupling
UserService has **no reference** to NotificationService. It doesn't know it exists. It publishes
an event to an abstraction (`IPublishEndpoint`). You could add a 4th service (say, an SMS service)
that also consumes `UserRegisteredEvent` — **without changing UserService at all**. This is the
open/closed principle at the system level.

### Pub/Sub vs point-to-point
MassTransit publishes to a RabbitMQ **exchange** (fan-out style). Each consumer type gets its own
**queue** bound to that exchange. So one event can fan out to many independent consumers — that's
**publish/subscribe**, not a point-to-point command.

## 2. Why these specific design choices

### Why an `IIdentityService` abstraction instead of using `UserManager` directly?
ASP.NET Core Identity's `UserManager<T>` has a huge constructor — it's painful to mock in unit
tests. By wrapping it in a small `IIdentityService` interface (implemented in Infrastructure),
the Application-layer handlers depend on a clean, mockable contract. The `RegisterCommandHandler`
test injects a `Mock<IIdentityService>` and verifies behavior with zero database.

### Why repositories in NotificationService but not a generic repo?
The `INotificationRepository` / `ITemplateRepository` interfaces keep the dispatch logic testable
(mock the repo, assert status transitions) and keep EF Core out of the Application layer. They're
**specific** (not a generic `IRepository<T>`) so each exposes only the queries actually needed —
avoiding the leaky-abstraction problem generic repositories often cause over EF.

### Why is the email a stub?
Three reasons: (1) no secrets/credentials in the repo, (2) it cleanly demonstrates the *integration
point* via `IEmailSender`, and (3) swapping in a real provider is a one-class change with no impact
on consumers or handlers. It signals you understand **seams** in a design.

### Why does `Shared.Contracts` contain only records, no logic?
If shared code contained business logic, services would become coupled through it — a change to
shared logic forces redeploying everyone, recreating the monolith problem. Sharing only the
**message contract** (the shape of the data) is the accepted boundary. Some teams go further and
don't even share that (each service redefines the event) to avoid *any* coupling — a valid
trade-off discussion to raise.

## 3. Security details

- **JWT signing:** HMAC-SHA256 with a symmetric secret key (`SymmetricSecurityKey`). The same key
  lives in UserService (to sign) and the Gateway (to validate). The token is validated for issuer,
  audience, lifetime, and signature.
- **Refresh token rotation:** every refresh issues a *new* refresh token and overwrites the old
  one. If an attacker steals a refresh token and uses it, the legitimate user's next refresh fails —
  detectable. (A production system would add reuse-detection/revocation lists.)
- **Refresh tokens are cryptographically random** (64 bytes from `RandomNumberGenerator`), not JWTs —
  they're opaque, stored server-side, and looked up in the DB.
- **Forgot-password doesn't leak account existence:** it returns `202 Accepted` whether or not the
  email exists, so an attacker can't enumerate registered emails.
- **Password reset tokens** are generated by Identity's `GeneratePasswordResetTokenAsync` (data-
  protection-backed, time-limited), carried in the event, and emailed as a link.

**Known simplification to be honest about:** the downstream services don't *re-validate* the JWT —
they trust the gateway. In a stricter zero-trust setup, each service would also validate the token
(defense in depth), because a service reachable on the internal network could otherwise be hit
directly. Worth mentioning proactively.

## 4. The testing strategy

| Test | What it proves | Technique |
|---|---|---|
| `RegisterCommandHandlerTests` | registers, publishes event, rejects duplicate | Moq on `IIdentityService`, `IJwtTokenService`, `IPublishEndpoint` |
| `LoginCommandHandlerTests` | tokens on valid creds, throws on wrong password | Moq |
| `RefreshTokenHandlerTests` | rotates token, rejects expired/unknown | Moq |
| `ConsumerTests` | consumer builds the right notification command | Moq `ConsumeContext<T>` |
| `NotificationDispatchServiceTests` | marks Sent after email, Failed on exception, retry increments | Moq repos + `IEmailSender` |
| `MassTransitHarnessTests` | full publish → consume flow, **no real broker** | MassTransit **in-memory test harness** |

The harness test is the standout: `AddMassTransitTestHarness()` spins up an in-memory bus,
publishes a real `UserRegisteredEvent`, and asserts the consumer actually ran — integration-level
confidence with unit-test speed.

> Note: the brief showed the legacy `InMemoryTestHarness` (MassTransit 7 API). This project is on
> MassTransit 8, where the harness is DI-based (`ITestHarness`). Know both names.

## 5. Build/runtime decisions worth defending

- **.NET 10 instead of the brief's .NET 8:** the installed SDK. Architecture is version-agnostic.
- **MassTransit pinned to 8.x, MediatR to 12.x:** their newer major versions (MassTransit 9,
  MediatR 14) moved to **commercial licenses**. The 8.x/12.x lines are the last free ones — a
  real-world dependency-licensing consideration.
- **Swashbuckle pinned to 7.2.0:** the 10.x line pulls Microsoft.OpenApi 2.0, which reorganized
  the API and broke the Swagger security-scheme config. Pinning kept the stable 1.6 API.
- **`FrameworkReference Microsoft.AspNetCore.App` in Infrastructure:** `AddDefaultTokenProviders()`
  lives in the ASP.NET Core *shared framework*, not the Identity NuGet package, so the class library
  needs the framework reference to see it.
- **Docker build context = repo root:** services reference `shared/Shared.Contracts`, which sits
  *outside* each service folder. A per-service build context couldn't reach it, so the Dockerfiles
  COPY relative to the root and compose sets `context: .`.
- **Gateway on host port 5080:** macOS reserves 5000 for AirPlay Receiver (Control Center).

## 6. Where this would need work for production (be ready to volunteer these)

- **No outbox pattern:** the user is saved and the event is published in two separate steps. If the
  app crashes between them, the event is lost. The **Transactional Outbox** pattern (MassTransit
  supports it) writes the event to the DB in the same transaction as the user, then a relay
  publishes it — guaranteeing at-least-once delivery. This is the #1 thing I'd add next.
- **No idempotency/dedup on consumers:** RabbitMQ is at-least-once, so a consumer can run twice.
  Production would dedupe by message id (e.g. store processed ids) to avoid double emails.
- **No retry/dead-letter policy tuning:** MassTransit has defaults; production would configure
  retry + redelivery + a dead-letter queue for poison messages.
- **Secrets in compose/appsettings:** the JWT key is in plaintext for the demo; production uses a
  secrets manager / environment injection.
- **SQLite for the demo:** fine for dev; production uses SQL Server/Postgres (already supported via
  the auto-detecting connection string).
- **No distributed tracing/correlation IDs** across the HTTP→broker hops; add OpenTelemetry.

## 7. Rapid-fire Q&A drill

**Q: What's a microservice vs a monolith?**
A: A monolith is one deployable with all logic together. Microservices split it into small,
independently deployable services, each owning its data and a single business capability.

**Q: Why not just have UserService call NotificationService over HTTP?**
A: That's tight, synchronous coupling — if Notification is down or slow, registration fails or
blocks. The broker decouples them: UserService fires and forgets; the message is durable and
delivered when Notification is ready.

**Q: What does MassTransit give you over raw RabbitMQ?**
A: A clean abstraction — typed publish/consume, automatic exchange/queue topology, retries,
serialization, and a test harness. You write `IConsumer<T>` instead of wiring channels and bytes.

**Q: How does the gateway know where to send a request?**
A: `ocelot.json` maps an upstream path template (what the client calls) to a downstream host/port
(the service). Ocelot matches the path and proxies the request.

**Q: How is the JWT validated at the gateway?**
A: `AddJwtBearer` with `TokenValidationParameters` checking issuer, audience, lifetime, and the
HMAC signature against the shared secret. Routes marked with `AuthenticationProviderKey: "Bearer"`
require a valid token; otherwise the gateway returns 401.

**Q: What happens if two `UserRegisteredEvent`s arrive for the same user?**
A: With no idempotency, two welcome emails. The fix is dedup by message/event id. Honest answer
that shows you know the at-least-once reality.

**Q: How do you add a new notification channel, say SMS?**
A: Add an `ISmsSender`, and either a new consumer on the same event or extend the dispatch service.
UserService doesn't change — that's the point of pub/sub.

**Q: Why CQRS / MediatR here, isn't it overkill?**
A: For a CRUD-only app, maybe. Here it keeps handlers single-purpose and trivially unit-testable,
and the pipeline behavior centralizes validation. It's a reasonable, common .NET pattern; I can
articulate the trade-off rather than cargo-culting it.

**Q: Where's the transaction boundary when registering?**
A: Within UserService's DB (create user + set refresh token). The cross-service "transaction"
(user + notification) is **not** atomic — it's eventually consistent via the event, which is the
correct microservices approach (saga/outbox for stronger guarantees).
