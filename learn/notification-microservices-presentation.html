🟢 Level 1 — Easy

The Big Picture, in plain language

The One-Sentence Pitch

"It's a notification platform built as microservices. When a user registers or resets their password in one service, a second service automatically sends them an email — and the two services never talk to each other directly. They communicate through a message broker, so they stay completely independent."

What problem does it solve?

In a monolith, the sign-up code and the email code are tangled together — if email breaks, sign-up breaks too. This project splits the work into 3 small, independent services, each with its own database, that can be developed, deployed and scaled separately.

The Three Services

ServiceIts only jobPort
UserServiceHandle accounts: register, login, JWT, password reset5001
NotificationServiceSend & record notifications (emails), store history + templates5002
API GatewayThe single front door — routes requests to the right service5080

Plus the plumbing

  • RabbitMQ — the "post office" that carries messages between services (UI on 15672).
  • Two databases — userservice.db and notificationservice.db. The gateway has none.

The Story of One Click 🔑

This is the flow you must memorize:

  1. A user clicks "Register".
  2. The request hits the API Gateway (the front door).
  3. The Gateway forwards it to UserService.
  4. UserService creates the account and shouts: "a user just registered!"
  5. That message goes into RabbitMQ.
  6. NotificationService is subscribed — it picks it up and sends a welcome email (faked: logged to console).
  7. It saves a record of that email in its own database.
You ──register──▶ Gateway ──▶ UserService ──"user registered!"──▶ RabbitMQ ──▶ NotificationService ──▶ 📧
Step 4 does not wait for step 6. UserService fires the message and moves on — this is asynchronous ("fire-and-forget"). The user gets a fast response; the email happens in the background.

Two Ways the Services Communicate

1. Synchronous (HTTP)

You → Gateway → UserService.
You ask, you wait, you get an answer.

Like a phone call. ☎️

2. Asynchronous (messages)

UserService → RabbitMQ → NotificationService. One drops a message and walks away; the other picks it up when ready.

Like sending a letter. ✉️

"The services talk two ways — HTTP through the gateway for direct requests, and asynchronous events through RabbitMQ for things that can happen in the background, like sending emails."

The Golden Rule: Each Service Owns Its Data

  • UserService has userservice.db — users, refresh tokens.
  • NotificationService has notificationservice.db — notification history, templates.
  • Neither service is allowed to touch the other's database.

If NotificationService needs the user's name, it does not reach into the user database — it gets that data from the message that was sent.

Database-per-service is the single most important microservices principle. It's what makes the services truly independent.

🟡 Level 2 — Medium

How it actually works under the hood

Clean Architecture — 4 Layers per Service

The rule: dependencies only point inward. Outer layers know inner layers, never the reverse.

Domain — core entities, no dependencies. AppUser, Notification, NotificationTemplate, enums
Application — business logic, CQRS handlers, interfaces, validators. RegisterCommandHandler, IIdentityService, INotificationService
Infrastructure — the "how": EF Core, Identity, repositories, email stub. AppDbContext, JwtTokenService, ConsoleEmailSender
API — entry point: controllers, DI wiring, middleware, Program.cs
"The Application layer defines interfaces like IIdentityService; Infrastructure implements them. That's Dependency Inversion — business logic depends on abstractions, not on the database. It's also what makes the handlers unit-testable with mocks."

CQRS with MediatR

Every operation is a Command (changes data) or a Query (reads data), each with its own handler. Controllers stay thin.

Controller → sends RegisterCommand → MediatR → RegisterCommandHandler runs the logic

Commands

  • RegisterCommand
  • LoginCommand
  • RefreshTokenCommand
  • UpdateProfileCommand

Queries

  • GetCurrentUserQuery
  • GetNotificationsQuery
  • GetUserNotificationsQuery

A MediatR pipeline behavior (ValidationBehavior) runs FluentValidation rules automatically before any handler — validation is cross-cutting, written once.

The Event Flow — Publishing

Events live in Shared.Contracts

A tiny shared library holding only the event shapes (plain C# records) — no business logic. Both services reference it so they agree on the message format.

public record UserRegisteredEvent {
    public string UserId { get; init; }
    public string Email { get; init; }
    public string FirstName { get; init; }
    public DateTime RegisteredAt { get; init; }
}

Publishing (UserService)

// inside RegisterCommandHandler, after creating the user
await _publishEndpoint.Publish(new UserRegisteredEvent { ... });

IPublishEndpoint (MassTransit) drops the message on RabbitMQ and returns immediately — UserService doesn't know who's listening.

The Event Flow — Consuming

public class UserRegisteredConsumer : IConsumer<UserRegisteredEvent> {
    public async Task Consume(ConsumeContext<UserRegisteredEvent> context) {
        // build a Welcome notification and send it
    }
}

When the event lands in RabbitMQ, MassTransit calls the matching consumer automatically.

How a Notification Gets "Sent"

The NotificationDispatchService (implements INotificationService) does the work:

  1. Look up the template for the type (e.g. Welcome) from the DB.
  2. Render it — replace {{FirstName}} placeholders with real data.
  3. Create a Notification record with status Pending, save it.
  4. Call IEmailSender.SendAsync(...).
  5. Success → mark Sent (+ timestamp). Throws → mark Failed (+ error).
  6. Save again.
Why a stub? ConsoleEmailSender just logs [EMAIL STUB] To: ... | Subject: .... No credentials needed, and the IEmailSender interface means you can swap in SendGrid/SES later without touching any caller.

The API Gateway (Ocelot)

The single entry point. Clients only ever talk to it; they never hit services directly. ocelot.json maps paths to downstream services:

/api/auth/*          → userservice:8080
/api/users/*         → userservice:8080          (requires JWT)
/api/notifications/* → notificationservice:8080  (requires JWT)
/api/templates/*     → notificationservice:8080  (requires JWT)

The gateway also validates the JWT before forwarding protected routes — an unauthenticated request gets a 401 at the gateway and never reaches the service.

Why a gateway? One public address, central auth, and clients don't need to know how many services exist or where they live.

Authentication: JWT + Refresh Tokens

  1. Login → UserService checks the password (ASP.NET Core Identity) and returns an access token (JWT, ~60 min) and a refresh token (~7 days, stored in the DB).
  2. The JWT is signed with a secret key — anyone can read it, nobody can forge it. It carries the user's id + email as claims.
  3. When the access token expires, the client calls /api/auth/refresh with the refresh token to get a new pair — and the old refresh token is rotated (replaced).
  4. Logout clears the refresh token so it can't be reused.

Databases & Docker

EF Core

  • Each service has its own DbContext + SQLite file.
  • On startup: MigrateAsync() auto-creates the schema, then seeds data (admin/test users; 3 templates).
  • Provider auto-detected from the connection string: SQL Server vs SQLite, zero code change.

Docker Compose

  • docker compose up --build starts RabbitMQ + 3 services on one network.
  • Services find RabbitMQ by hostname rabbitmq.
  • depends_on + healthcheck make services wait until RabbitMQ is truly ready.
"Each service migrates and seeds its own database on startup, so a fresh docker compose up gives you a working system with seed accounts and templates — no manual setup."

🔴 Level 3 — Hard

Trade-offs, deep concepts & the Q&A drill

Eventual Consistency

When a user registers, the account exists immediately, but the welcome notification appears a moment later (after the event is consumed). The two databases aren't updated in one atomic transaction — the system is eventually consistent.

Q: What if NotificationService is down when the event is published? RabbitMQ holds the message in the queue. When the service comes back, MassTransit delivers it and the consumer runs — no notification is lost. That's the durability benefit of a broker over a direct HTTP call, where a downstream outage could fail or block registration.
You trade immediate cross-service consistency for independence and resilience. That's the core microservices bargain.

Loose Coupling & Pub/Sub

UserService has no reference to NotificationService. It publishes to an abstraction (IPublishEndpoint) and doesn't know who listens.

Extensibility: You could add a 4th service (e.g. SMS) that also consumes UserRegisteredEvent — without changing UserService at all. That's the open/closed principle at the system level.

MassTransit publishes to a RabbitMQ exchange (fan-out); each consumer type gets its own queue bound to it. So one event can fan out to many independent consumers — that's publish/subscribe, not point-to-point.

Why These Design Choices

Why an IIdentityService abstraction instead of UserManager directly?

UserManager<T> has a huge constructor — painful to mock. Wrapping it in a small interface keeps handlers depending on a clean, mockable contract, so tests run with zero database.

Why specific repositories, not a generic IRepository<T>?

They keep EF Core out of the Application layer and expose only the queries actually needed — avoiding the leaky-abstraction problem generic repos cause over EF.

Why does Shared.Contracts have no logic?

Shared logic re-couples services through it — a change forces redeploying everyone, recreating the monolith. Sharing only the message contract is the accepted boundary.

Security Details

  • JWT signing: HMAC-SHA256 with a symmetric secret; validated for issuer, audience, lifetime & signature.
  • Refresh token rotation: every refresh issues a new token and overwrites the old — a stolen token is detectable on the next legitimate refresh.
  • Refresh tokens are opaque random bytes (64 bytes from RandomNumberGenerator), stored server-side — not JWTs.
  • Forgot-password returns 202 regardless of whether the email exists — no account enumeration.
  • Reset tokens come from Identity's data-protection-backed, time-limited generator.
Downstream services trust the gateway and don't re-validate the JWT. A stricter zero-trust setup would have each service validate too (defense in depth). Worth volunteering proactively.

Build / Runtime Decisions Worth Defending

  • .NET 10 instead of .NET 8 — the installed SDK; architecture is version-agnostic.
  • MassTransit pinned to 8.x, MediatR to 12.x — newer majors (MT 9, MediatR 14) moved to commercial licenses; these are the last free lines. A real dependency-licensing call.
  • Swashbuckle pinned to 7.2.0 — the 10.x line pulls Microsoft.OpenApi 2.0 which broke the Swagger security config.
  • FrameworkReference in Infrastructure — AddDefaultTokenProviders() lives in the ASP.NET Core shared framework, not the NuGet package.
  • Docker build context = repo root — services reference shared/, outside their own folder.
  • Gateway on host port 5080 — macOS reserves 5000 for AirPlay Receiver.

What I'd Add for Production

Volunteer these — they show maturity:

  • Transactional Outbox: today the user save and the event publish are two steps; a crash between them loses the event. The outbox writes the event in the same DB transaction, then relays it — at-least-once delivery. My #1 next step.
  • Idempotency / dedup on consumers: RabbitMQ is at-least-once, so a consumer can run twice → dedupe by message id to avoid double emails.
  • Retry + dead-letter policy for poison messages.
  • Secrets manager instead of plaintext JWT key.
  • SQL Server/Postgres in prod (already supported via the auto-detecting connection string).
  • Distributed tracing (OpenTelemetry) across the HTTP → broker hops.

Rapid-Fire Q&A — Part 1

Microservice vs monolith?A monolith is one deployable with all logic together. Microservices split it into small, independently deployable services, each owning its data and one business capability.
Why not just have UserService call NotificationService over HTTP?That's tight, synchronous coupling — if Notification is down or slow, registration fails or blocks. The broker decouples them: fire-and-forget, durable, delivered when ready.
What does MassTransit give you over raw RabbitMQ?A clean abstraction: typed publish/consume, automatic exchange/queue topology, retries, serialization, and a test harness. You write IConsumer<T> instead of wiring channels and bytes.

Rapid-Fire Q&A — Part 2

How does the gateway know where to send a request?ocelot.json maps an upstream path template to a downstream host/port. Ocelot matches the path and proxies the request.
What if two UserRegisteredEvents arrive for the same user?With no idempotency, two welcome emails. The fix is dedup by event id — an honest answer that shows you know the at-least-once reality.
Where's the transaction boundary when registering?Within UserService's DB. The cross-service "transaction" (user + notification) is not atomic — it's eventually consistent via the event, which is the correct microservices approach. Saga/outbox for stronger guarantees.
